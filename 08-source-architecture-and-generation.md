# Source Architecture and Generation Alignment

## Scope

This document records additional findings derived from:

- the Linux Foundation LFS114 chapter on exploring the free5GC source code;
- local SMF `main` at `df5818d`
- the current `ees-only` implementation at `8668372`

It intentionally excludes issues already covered by the other reports, such
as the oversized EES handler, missing Nupf consumer abstraction, duplicated
route segment, static UPF endpoint, and local protocol model duplication.

## Main-Branch Reference Flow

The local main branch starts SMF through the following ownership chain:

```text
cmd/main.go
  -> factory.ReadConfig()
  -> service.NewApp()
     -> context.Init()
     -> consumer.NewConsumer()
     -> processor.NewProcessor()
     -> sbi.NewServer()
  -> SmfApp.Start()
     -> NRF registration
     -> SBI HTTP/2 server
     -> PFCP server
```

This structure matters because configuration, runtime state, outbound clients,
business logic, and HTTP exposure each have an explicit owner and lifecycle.

## High: A Generated File Is Used as a Handwritten Implementation File

`internal/sbi/api_eventexposure.go` in main is an OpenAPI-generated route and
handler skeleton. Its header explicitly says:

```text
Generated by: OpenAPI Generator
```

The ees-only branch turns that same file into a large handwritten
implementation containing:

- local request and response models;
- validation;
- subscription orchestration;
- error mapping;
- logging helpers;
- UPF request mapping;
- Location construction.

This creates an ownership conflict even though Git does not report one. If the
SMF API skeleton is regenerated from a newer specification, the handwritten
EES implementation can be overwritten or produce a very large generated-code
diff.

### Recommended Direction

Choose and document one ownership model:

1. Keep `api_eventexposure.go` generated and limit it to route/HTTP adaptation,
   then place custom logic in processor and consumer files.
2. Stop treating the file as generated, remove the generated-code claim, and
   accept full manual maintenance.

The first option matches the free5GC source structure more closely.

Generated operation names such as `CreateIndividualSubcription` contain the
upstream specification's spelling. Keep those names at the generated boundary,
but use correctly spelled domain method names inside the processor and
application logic.

## Medium: Comments Describe Patch History Instead of Stable Invariants

The EES implementation contains detailed comments, but some are tied to the
local patch sequence rather than stable behavior. For example, the Create
handler still contains comments referring to earlier patch behavior even though
UPF cascading now exists.

This is not a lint failure, but it is not ideal for upstream review. Comments
in free5GC-facing code should describe:

- protocol invariants;
- ownership boundaries;
- concurrency assumptions;
- non-obvious lifecycle policy.

They should avoid historical notes such as `Patch 1`, temporary design
explanations, or comments that become stale when later patches change the
behavior.

### Recommended Direction

For the clean PR branch, keep comments concise and stable:

1. leave generated files as generated route adapters;
2. move policy comments into processor/consumer/repository code;
3. remove patch-history comments from runtime files;
4. keep longer rationale in documentation or tests.

## High: No Reproducible OpenAPI Generation Workflow

The branch stores authoritative-looking specifications under `docs/spec`, but
the repository does not define:

- the OpenAPI Generator version;
- the exact generation command;
- generator options and templates;
- output package ownership;
- post-generation patches;
- a CI check for generated-code drift.

The training material treats the specification and generation command as the
repeatable source of the models and client interfaces. Copying YAML files into
the repository without a reproducible generation path does not provide that
guarantee.

This is especially important for Task2 because free5GC OpenAPI `v1.2.3`
contains an Nsmf Event Exposure client and `models.NsmfEventExposure`, but does
not contain the reviewed `UPF_EVENT` extension fields or a Nupf Event Exposure
client.

### Recommended Direction

Add a versioned generation manifest or script that records:

```text
input specification revision
OpenAPI Generator image/JAR version
generator name
additional properties
output directories
formatting command
known patches or schema overlays
```

Generate into a dedicated protocol/client package rather than mixing generated
and handwritten code in the same file.

Add a CI job that regenerates the selected API packages and fails when the
working tree changes.

## Medium: Generated Model Provenance Is Ambiguous

The `models.NsmfEventExposure` available in free5GC OpenAPI `v1.2.3` has a
generated header referring to `Nnwdaf_DataManagement` and TS 29.520, even
though the model represents an Nsmf Event Exposure resource.

The same model also lacks the Task2 fields required by the local contract.

This means that importing a generated type is not by itself proof that the
type came from the intended TS 29.508 revision. The specification source,
release, schema composition, and generated module version must all be tracked.

### Recommended Direction

Maintain a compatibility matrix containing:

| Interface | Authoritative TS | Required revision | Generated package | Known extension |
| --- | --- | --- | --- | --- |
| Nsmf Event Exposure | TS 29.508 | Project-selected revision | `smf/EventExposure` | Task2 UPF event fields |
| Nupf Event Exposure | TS 29.564 | Project-selected revision | To be generated | Outer `subscription` wrapper |
| ProblemDetails | TS 29.571 | Matching common-data revision | `models` | None |

Tests should assert the required JSON field names against this matrix instead
of assuming that any available `models.NsmfEventExposure` is sufficient.

## Medium: EES State Has No Explicit Application Lifecycle Owner

The training material identifies `internal/context` as the NF runtime-state
center. Main initializes context as part of `service.NewApp`, and the app owns
consumer, processor, server, cancellation, and shutdown.

The EES subscription store is instead a package-level singleton initialized
implicitly when its package is loaded. It is not:

- a field of `SMFContext` or `SmfApp`;
- constructed by `service.NewApp`;
- supplied to the processor;
- closed, drained, or reconciled during termination;
- replaceable in tests without manipulating global state.

Restart and orphan behavior is documented elsewhere. The additional
architectural issue is the absence of explicit ownership.

### Recommended Direction

Define a subscription repository interface and create its implementation in
the application construction path:

```text
service.NewApp
  -> create EES repository
  -> inject repository into processor
  -> expose shutdown/reconciliation hook
```

For V0 the implementation may remain an in-memory map with a mutex. Explicit
ownership improves test isolation and leaves a controlled replacement point
for durable storage.

Avoid placing consumer or transport objects in `internal/context`; context
should own runtime data, while outbound communication remains in
`internal/sbi/consumer`.

## Medium: Direct Nupf HTTP Calls Bypass Standard SBI Metrics

Main's generated consumers configure free5GC's outbound metric hook:

```go
configuration.SetMetrics(sbi_metrics.SbiMetricHook)
```

The EES Nupf client creates a plain `http.Client` and performs requests
directly. Consequently, Nupf Create/Delete calls do not automatically
participate in the standard outbound SBI measurements used by other SMF
consumers.

This reduces operational visibility and makes Nupf failures harder to compare
with NRF, UDM, PCF, AMF, or CHF calls.

### Recommended Direction

When a generated Nupf client is available, configure it with the same
free5GC SBI metric hook as existing consumers.

If a temporary manual transport remains necessary, wrap it with equivalent
metrics for:

- target service and operation;
- request count;
- response status;
- latency;
- transport failures;
- retries.

Metrics should not include SUPI, UE IP, notification URI, or subscription IDs
as labels because those values have high cardinality and may contain sensitive
information.

## Medium: Service Advertisement Is Not Tied to EES Readiness

Main uses `serviceNameList` for two related startup behaviors:

1. register the service in the NF profile;
2. expose its routes in `internal/sbi/server.go`.

With ees-only, `nsmf-event-exposure` can be enabled while no relevant UPF has a
usable Nupf API root. SMF then advertises and exposes EES successfully, but
every valid Create request for those sessions fails only at runtime.

The factory validates syntax, but not the complete capability dependency:

```text
Nsmf EES enabled
  -> appropriate UPF topology exists
  -> Nupf endpoint or discovery path is available
  -> required EES configuration is valid
```

### Recommended Direction

Add an EES readiness validation phase during application construction.

For a strict deployment mode, startup should fail when EES is enabled but its
required dependencies are invalid.

For dynamic or partially capable topologies, startup may continue, but the NF
profile and readiness status should accurately describe the available
capability. A UPF without EES support must remain valid for ordinary PDU
sessions; it should not be treated as globally invalid.

## Low: Route Registration Silently Ignores Unsupported Methods

Main's `applyRoutes` dispatches known method strings through a switch and has
no default error. A generated or handwritten route with an unsupported or
misspelled method is silently omitted during startup.

For a service whose implementation is assembled from generated route
descriptors, silent omission can make the NF profile advertise an operation
that the router never registered.

### Recommended Direction

Validate route descriptors during server construction:

- reject unsupported HTTP methods;
- detect duplicate method/path pairs;
- identify duplicate route names;
- log the final registered service, method, and path set.

Return an initialization error rather than allowing a partially registered
service to start unnoticed.

## Low: EES Startup and Shutdown Are Not Observable as a Capability

The main startup log reports the SBI listener, but EES does not emit a concise
capability summary showing:

- whether Nsmf Event Exposure is enabled;
- which V0 feature subset is active;
- whether Nupf transport is statically configured or discoverable;
- how many configured UPFs expose an EES endpoint;
- whether the repository is volatile.

Individual request logs exist, but they do not replace startup-level
configuration observability.

### Recommended Direction

Emit one structured startup summary without sensitive values, for example:

```text
service=nsmf-event-exposure
enabled=true
mode=single-supi-periodic-per-session
repository=in-memory
nupf_endpoint_source=per-upf-config
ees_capable_upfs=2
```

Emit a corresponding shutdown summary including the number of local
subscriptions still present and any cleanup/reconciliation action taken.

## Suggested Implementation Order

1. Define generated-versus-handwritten file ownership.
2. Add a reproducible OpenAPI generation manifest and drift check.
3. Track specification and generated-model provenance.
4. Move EES repository construction into the application lifecycle.
5. Integrate Nupf calls with standard SBI metrics.
6. Add EES readiness validation before NRF registration.
7. Validate route descriptors during server construction.
8. Add capability-level startup and shutdown logs.

These changes are architectural and operational improvements. They should be
implemented without changing the fixed V0 subscription semantics.
